## 3. Design Patterns

### מלכודת: Bridge מול Facade
מקור: `sample_test.pdf` (tier-7, ביטחון-נמוך, אך תבנית טובה): שאלת Facade עם אופציית-הסחה שמתארת בעצם Bridge ("connection between component hierarchy and its implementation").
**זהו: Facade מסתיר מורכבות (ATM); Bridge מפריד שני צירי שינוי עצמאיים.**

### מלכודת: Bridge מול Strategy
טבלת השוואה מלאה נמצאה ב-lab_10 (Lab 10) — שתי התבניות משתמשות ב-Composition ונראות דומות בקוד:
- Bridge = Structural, מפריד Abstraction מ-Implementation, הלקוח לרוב לא בוחר מפורשות.
- Strategy = Behavioral, בוחרת אלגוריתם בזמן ריצה, הלקוח בוחר מפורשות.

### מלכודת: תרשים ThreadScheduler (Preemptive/TimeSliced x Unix/Windows)
נבדק ב-2018 (שתי סריקות סטודנטים — ניתנו תשובות **שונות ולא-ודאיות**: Factory באחת, Bridge בסימון-לא-ודאי בשנייה) וב-2024 (**מאושר 100% נכון: Bridge**).
**מסקנה: הפתרון האותנטי הוא Bridge** — שני צירי שינוי בלתי-תלויים (מדיניות תזמון x מערכת הפעלה) הם הדוגמה הקלאסית ל-Bridge, וזה גם מאושר ע"י המקור המהימן ביותר (student_solution_2024, ציון 75/100 עם breakdown רשמי).

### מלכודת: Observer — "מה מהבאים אינו נכון"
זו השאלה העקבית ביותר בקטלוג כולו. נבדקה ב-4 מבחנים שונים (2018 פעמיים, 2019 שחזור, קונספט חוזר). תשובת ה"אינו נכון" (=זו שהיא נכונה לשאלה) היא **תמיד**:
> "ה-Observable חייב להיות מודע למחלקות הקונקרטיות של ה-Observers השונים, ולפיכך היחסים בינו לבין ה-Observers אינם אבסטרקטיים."

זה שגוי — הקשר בין Observable ל-Observers הוא **תמיד אבסטרקטי** (דרך ממשק Observer בלבד); ה-Observable לא צריך להכיר את המימושים הקונקרטיים.

### מלכודת: Mediator מול Adapter
מקור: lec_9 slide 23-25 בפירוש. **Adapter** — כשכבר יש שני בסיסי-קוד קיימים שצריך לחבר. **Mediator** — כשמתכננים ארכיטקטורה חדשה או עושים ריפקטורינג.

### מלכודת: Decorator מול ירושה
מקור: presentation8. ירושה קובעת התנהגות לכל המחלקה בזמן קומפילציה; Decorator משנה מופע ספציפי בזמן ריצה בלבד, בלי להשפיע על שאר המופעים.

### מלכודת: Class Adapter מול Object Adapter
מקור: lab3_upd Lab Exercise 2/3, atp2021_lab3. אם ננסה לממש גם `ITask` וגם `IWorkerSorter` על אותה מחלקה (Solution 1) — זו הפרת SRP ("A BubbleWorkerSorter is not a Task! It should stay a pure sorter"). Class Adapter (יורש) עובד רק ל-BubbleSort ספציפי; **Object Adapter (מכיל) עובד לכל sorter** — עדיף.

## 4. Generics, Collections, equals/hashCode

### מלכודת: "בדיחת ה-GenericStack"
נבדקה כמעט מילה-במילה ב-2018 (שלוש סריקות: שני סטודנטים + שחזור), 2019, ו-2024 (Germany/2024 במקום Russia/2018).
תשובות מסיחות: "שגיאת הידור", "שגיאה בזמן ריצה".
**התשובה הנכונה: אין שגיאה כלל — הפלט הוא שתי המילים/המספרים יחד** (למשל "Rusia 2018" או "Germany 2024"), כי אלו שני מופעים גנריים נפרדים לחלוטין (`GenericStack<String>` ו-`GenericStack<Integer>`).

### מלכודת: equals() בלי hashCode()
נבדקה ב-4 שחזורים שונים בעקביות מוחלטת (2018 x2, 2019, 2024/lab-context).
תשובת מלכודת: "זה בכלל לא אפשרי — שגיאת הידור אם דורסים אחת בלי השנייה".
**התשובה הנכונה:** זה מותר קומפילטיבית! אבל יגרום שאובייקטים "זהים" מבחינה לוגית (`equals()==true`) יקבלו `hashCode()` שונה, מה שישבור התנהגות תקינה באוספים מבוססי-hash (HashMap/HashSet — ייתכנו כניסות כפולות לאותו "מפתח לוגי").

### מלכודת: String Pool ו-Integer Caching
מקור: `advancedprogramming/lecture_1_2/Main.java` (קוד אמיתי מהמעבדה):
```java
String s1 = new String("moshe");  String s2 = new String("moshe");
String s3 = "moshe";               String s4 = "moshe";
s1 == s2 // false! שני אובייקטים נפרדים ב-Heap
s3 == s4 // true! שני literals משותפים מה-String Pool
```
```java
Integer a1 = 5; Integer a2 = 5;
a1 == a2 // true - Integer caching (-128 עד 127) - אבל שביר מחוץ לטווח!
```
**מסקנה כללית: תמיד השתמשו ב-`.equals()` להשוואת אובייקטים/Wrapper types, לעולם לא ב-`==`.**

### מלכודת (מ-sample_test.pdf, tier-7, ביטחון נמוך): hashCode פגום מבוסס אורך בלבד
מקור: `grade_100_term_a` Q7 (2025, מאושר): מחלקת Person עם `hashCode(){ return name.length(); }` — הבעיה המדויקת: "הפונקציה מתחשבת רק באורך השם, ולכן ייתכנו קודים זהים לשמות שונים באותו אורך" (לא "תמיד תחזיר 0" ולא "לא תואמת equals" כשלעצמה — התשובה המדויקת עוסקת בהתנגשויות מיותרות עקב חוסר בידול).

## 6. Threads ובטיחות תהליכונים

### מלכודת: obj.run() ואז obj.start()
נבדקה ב-3 שחזורים שונים (2018 x2, 2019) בקוד זהה כמעט:
```java
public class Test extends Thread implements Runnable {
    public void run() { System.out.print("GFG "); }
    public static void main(String[] args) throws InterruptedException {
        Test obj = new Test();
        obj.run();   // סינכרוני!
        obj.start(); // thread חדש שגם קורא ל-run()
    }
}
```
תשובות מסיחות: "שגיאת ריצה", "שגיאת הידור", "אף תשובה אינה נכונה".
**התשובה הנכונה: GFG GFG** (מודפס פעמיים — פעם סינכרונית מ-run(), פעם מ-thread חדש שנוצר ע"י start()).

### מלכודת: volatile מבטיח אטומיות?
מקור: `sample_test.pdf` (tier-7) בשילוב lec_8/lec_9 (tier-1): "volatile keeps a private copy per thread" הוא **המשפט השגוי** (ההפך נכון — volatile *מונע* עותק פרטי, ומכריח קריאה מה-Main Memory). **volatile מבטיח נראות (visibility) בלבד, לעולם לא אטומיות (atomicity)** — `counter++` על שדה volatile עדיין לא thread-safe.

### מלכודת: מהו Deadlock
תשובות מסיחות: "thread מסיים ונעל", "thread מושהה", "מצב בו thread סיים נעילתו ומת".
**התשובה הנכונה: שני threads (או יותר) לא מתקדמים כי הם ממתינים זה לזה לשחרור משאב.**
מקור: date_2025 Q16 (מאושר).

### מלכודת: Singleton לא Thread-Safe
מקור: lec_8 slide 14 (מפורש): מימוש נאיבי `if(instance==null) instance=new X();` **אינו** thread-safe — שני threads יכולים לעבור את הבדיקה בו-זמנית וליצור שני מופעים, "שובר" את הבטחת ה-Singleton.

### מלכודת: static counter משותף בין threads
מקור: grade_100_term_a Q19 (2025, מאושר). שני threads מריצים `counter++` על שדה **static** משותף. תשובה נכונה: **בדיוק 10 שורות** יודפסו סה"כ, הערכים אינם יורדים (non-decreasing) אך **סדר ההדפסה בין ה-threads אינו צפוי מראש** — לא "5 ואז 5 בנפרד" ולא "1 עד 10 מסודר לחלוטין".
